An Approach to Producing Clean Code
Nalla Senthilnathan <nsenthil@hotmail.com>

 


Introduction: This document describes an approach to documenting requirements that in-turn leads to producing clean code. The proposed approach is illustrated for a sample web application. Advantages of the approach over the conventional approach are also discussed.

A requirements document used by the web application development teams generally is a use case document. Traditional use case docs typically are verbose with sections like:

Description:
Actors:
Pre-Conditions:
Post-Conditions:
Use Case Diagram:
Happy Path Event Steps:
Alternate Path Event Steps:
Exception Path:
Business Rules:
etc.


Having worked with such a format for more than a decade I have noticed several limitations – the document looks nice and clean when the project starts. However, as the project progresses with iterations, the document gets cluttered with requirement fragments misplaced in various sections. Furthermore, the teams often add on UI requirements into the same document making it even more cluttered. A cluttered requirements document often leads to cluttered code development. Traceability and testability of the requirements and business rules becomes a drag on the project timeline.

I propose a new format below that I found solves many of the limitations discussed above.

It is interesting to note the following features of the above table:

1.       It includes all the elements of a typical use case document in a compact space.
(Note that alternate paths can be incorporated into the above table like replacing processPaymentBR with processCreditCardPaymentBR and processPayPalPaymentBR etc.)

2.       It also includes the business rule names and data rule names nicely tucked in within the same compact space!

3.       The business rule names and data rule names are hyperlinked for easy navigation. This feature keeps the table uncluttered.

4.       The event steps are demarcated into rows in such a way that each row’s content can be developed tested and deployed independent of the other rows.
This feature should appeal to agile scrum/sprint aficionados. This feature should also appeal to project managers since the rows give clear guidance for resource requirements

5.       Each row of the event steps’ demarcation is typically associated with a screen in a web application. In the above table such screen specifications are hyperlinked and the details presented in a separate table.

6.       When new requirements are added in later iterations of the project it is easy to see that the complexity of the document should remain the same as that of the first iteration.

The only change in the above table is the addition of a fifth column - Unit-Testable Services . In this column the team lead identifies the services that implement each event step’s business and data rules. The services are grouped (in four rows in the above example) for incremental develop, test and deploy.

It is now easy to see that the above features naturally lead to writing clean code for the developers. The identified services satisfy some of the core principles of clean code like:

1.       Unit Testability – each service is unit testable by design

2.       Loose coupling – one service does not depend on another service while unit-testing

3.       Meaningful/readable names – service names directly correspond to event steps or business rules

4.       Incremental evolution – if a business rule or data rule changes then only one service need to change. All other services remain unaffected. If a new rule is added only one service need to be updated or a new service created. So the system can grow from simple to complex incrementally.

Conclusions: In conclusion it can be mentioned that the proposed approach has the following advantages:

1.       Produces a clean requirements document whose complexity remains the same through the project iterations

2.       Incorporates agile scrum/sprint principles naturally by facilitating incremental develop-test-deploy cycles

3.       Provides clear guidance for the project managers in estimating resource requirements

4.       Provides clear guidance for the tech leads in estimating project timelines

5.       Provides clear guidance for the developers in writing clean code

6.       Traceability and testability of the requirements and business rules are readily apparent
